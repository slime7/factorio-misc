# 设计与维护

## 核心思路

这版蓝图不再使用“固定延时防抖”，而是采用回差控制：

- 低阈值：决定什么时候可以启动
- 高阈值：决定什么时候可以释放

这样做的目标不是“等几秒再切”，而是“开始补了就先补到一段安全库存再切”。

## 逻辑分段

1. 低阈值减当前库存，得到启动缺口
2. 只保留缺口大于 `0` 的候选
3. 如果某个信号已经见底，则给它更高的抢占优先级
4. 从候选里只选一个，默认取缺口最大的那个；但见底信号可以打断当前锁存
5. 若当前未锁存，则写入新候选
6. 若当前已锁存且对应库存还没到高阈值，则给予中等保持补偿
7. 若别的信号缺口明显更大，仍可提前抢占当前锁存
8. 一旦达到高阈值，释放锁存，重新选择

## 可调参数

- 低阈值：如 `2000`
- 高阈值：如 `4000`
- 选择策略：当前为“低阈值缺口最大优先；当前锁存只有中等保持补偿；见底信号可强制抢占”

## 为什么要加“见底抢占”

如果被监控的多个信号之间存在依赖关系，例如：

- `钢材`
- `熔融铁`

那么制造钢材的过程中可能先把熔融铁耗空。  
如果仍然死守“锁存后一定要到高阈值才释放”，系统就会一直卡在钢材上，反而补不回缺口已经明显扩大的熔融铁。

因此这版逻辑额外加入一条规则：

- 正常情况下仍然遵循回差锁存
- 但当前锁存只拿到中等保持补偿，而不是绝对优先级
- 只要别的信号缺口明显更大，就可以提前打断当前锁存
- 如果某个被监控信号已经见底，它还会获得更高优先级，直接抢占当前锁存

这样可以避免下游配方把上游原料耗尽后出现自锁。

## 为什么不内置配方映射

有些场景里：

- 监控信号是流体或物品库存
- 机器真正需要的是另一条明确的配方信号

这两者不一定一一对应。  
因此这版蓝图只负责“选中并稳定输出某个原始信号”，把“原始信号 -> 明确配方信号”的映射留给右侧后级逻辑处理。

## 显示器端点

输入端和输出端都改成了 `display-panel`：

- 用文字直接标明接线用途
- 用图标让端点更容易识别
- `always_show = true`，尽量减少只靠 Alt-mode 才能读懂结构的情况
- `control_behavior.parameters` 里直接写说明文字、图标和 `signal-1 = signal-1` 的恒真条件
- 不再依赖辅助常量组合器

## 通用脚本

当前蓝图的真源放在同目录的 `蓝图.json`。  
`.agents/skills/factorio-blueprints/scripts/` 下的脚本只负责通用的发现、加载、`json/jsonc` 解析、编解码和构建，不再内嵌具体蓝图逻辑。

内置到蓝图里的 `description` 需要尽量简短。  
按当前实测，中文说明在导入游戏后会表现出接近 `500` 字节的截断现象，因此更完整的使用说明应放在 `README.md`。

列出已登记蓝图：

```powershell
python .agents/skills/factorio-blueprints/scripts/build_blueprint.py list
```

重建当前蓝图：

```powershell
python .agents/skills/factorio-blueprints/scripts/build_blueprint.py build recipe-signal-switch
```

解码成 JSON 查看：

```powershell
python .agents/skills/factorio-blueprints/scripts/factorio_blueprint_codec.py decode "blueprints/配方信号阈值锁存开关/蓝图.txt" --pretty
```
